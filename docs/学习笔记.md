# AI 企业知识库助手学习笔记

## 1. 项目要解决什么问题

普通大模型不知道企业私有文档的内容。RAG 的做法不是重新训练模型，而是在回答前先从企业知识库中找出相关片段，再把片段交给大模型生成答案。

```text
上传文档 → 解析 → 清洗 → 切片 → Embedding → pgvector
用户提问 → 问题 Embedding → 相似度检索 → 拼接上下文 → 模型回答 + 引用
```

## 2. C# 与当前 Python 技术栈对照

| C# / ASP.NET Core | Python 项目 | 作用 |
|---|---|---|
| Controller / Minimal API | FastAPI Router | 定义 HTTP 接口 |
| Request/Response DTO | Pydantic Model | 校验输入和输出 |
| Interface | `Protocol` | 定义解析器、模型提供者契约 |
| EF Core `DbContext` | SQLAlchemy `AsyncSession` | 查询、跟踪和提交数据 |
| EF Core Migration | Alembic | 管理数据库结构版本 |
| ASP.NET Core DI | FastAPI `Depends` | 创建请求级依赖 |
| BackgroundService | FastAPI 后台任务（第一阶段） | 执行文档处理 |

## 3. 阶段 1A：项目基础与数据库

1A 建立了 FastAPI 应用工厂、统一配置、`request_id` 和错误协议，并使用 PostgreSQL + pgvector 保存业务数据。

四张核心表：

- `knowledge_bases`：知识库容器。
- `documents`：上传文件及处理状态。
- `ingestion_jobs`：文档处理任务及失败阶段。
- `document_chunks`：切片正文、来源信息和 512 维向量。

关键理解：`AsyncSession` 类似一次请求内使用的 `DbContext`，它表示有状态的事务边界，不能在多个并发任务间共享。

## 4. 阶段 1B：知识库与文档解析

1B 支持创建/列出知识库，安全上传 PDF、DOCX、XLSX、Markdown 和 TXT，并查询文档状态。

安全规则包括：

- 扩展名白名单和 20 MB 上限。
- 原文件名只展示，磁盘保存名使用 UUID。
- SHA-256 用于同知识库文件去重。
- 解析器统一返回 `ParsedSection`，保留 PDF 页码、Excel 工作表/行号和 Markdown 标题。

解析器只负责“文件 → 结构化文本”，不直接访问数据库，也不生成向量。这种边界类似 C# 中把文件适配器与应用服务分开。

## 5. 阶段 1C：清洗、切片与向量入库

1C 完成数据清洗、中文递归切片、Embedding 和向量入库：

1. 规范换行和空白，丢弃空段落。
2. 默认按 800 字切片，相邻切片重叠 120 字。
3. 用 `EmbeddingProvider` 把文本转换为浮点向量；1D 接入 BGE 后使用 512 维。
4. 在一个事务中写入全部 `DocumentChunk`；任何一步失败都不能留下半套切片。

Embedding 可以理解为把文本映射到“语义坐标”。含义相近的文本，其向量距离通常更近。

自动化测试使用确定性的 `FakeEmbeddingProvider`，不会访问网络或产生费用；本地运行使用 `BAAI/bge-small-zh-v1.5` 生成真实语义向量。上传后的后台任务按 `parsing → embedding → store` 处理，全部切片成功后才在一个事务中写入；失败时文档状态变为 `failed`，不保留半套切片。

## 6. 阶段 1D：检索、问答与引用

1D 使用固定的两阶段 RAG：

```text
Retrieve 检索 → Generate 生成
```

问题和文档切片都由同一个本地 BGE Small 转换为 512 维归一化向量。`VectorRetriever` 把 SQLAlchemy 查询封装成一个 `search()` 方法，强制使用 `knowledge_base_id` 隔离数据，并通过 pgvector 的余弦距离返回 Top K 片段。

BGE v1.5 的短问题检索需要在 Query 前添加官方检索指令“为这个句子生成表示以用于检索相关文章：”，文档片段不添加。这个差异封装在 `LocalEmbeddingProvider.embed_query()` 内，API 和业务服务不需要关心模型细节。

余弦距离越小越相近，接口使用 `relevance_score = 1 - cosine_distance` 转换为“越大越相关”的分数。第一版默认阈值是 0.55；没有片段达到阈值时直接拒答，不调用 DeepSeek。

`RagService` 类似 C# 的 Application Service，负责串联 Embedding、Retriever、Prompt 和 ChatProvider；`VectorRetriever` 类似 Repository 或 Query Service。DeepSeek 只看到带 `[1]`、`[2]` 编号的上下文，最终文件名、页码和正文由后端映射，不能让模型自由编造来源。

本地使用 `deepseek-v4-flash` 生成中文答案。Key 只保存在被 Git 忽略的 `.env`；自动化测试改用 `FakeChatProvider`，保证稳定且无费用。

## 7. 自测题

1. 为什么原始文件名不能直接作为服务器保存路径？
2. `Document` 和 `IngestionJob` 为什么要分成两张表？
3. 为什么解析器不应该直接写数据库？
4. 切换 Embedding 维度后，为什么旧向量需要重新生成？
5. 为什么检索必须强制带 `knowledge_base_id`？
6. 为什么无相关片段时不应该调用大模型？
7. 为什么引用来源必须由后端映射，而不能直接相信模型？

能不用看答案讲清这些问题，就理解了第一版 RAG 后端的核心闭环。
