# 项目一键启动设计

## 1. 目标

为 Windows 用户提供两种等价的简单入口：双击 CMD 文件，或在 PowerShell 中执行一条命令。启动器负责检查运行前置、启动完整 Docker Compose、等待统一 gateway 就绪并打开浏览器；停止器只安全停止本项目容器并保留持久数据。

## 2. 已确认范围

- 根目录新增 `启动项目.cmd` 和 `停止项目.cmd`，供不熟悉命令行的用户双击使用。
- 新增 `scripts/start-project.ps1` 和 `scripts/stop-project.ps1`，作为可测试、可复用的核心实现。
- 启动完整演示拓扑 `deploy/docker-compose.yml`，浏览器入口固定为 `http://127.0.0.1:8080`。
- 启动器不自动创建 `deploy/.env`、不写入随机密钥、不创建管理员账号；缺少配置时只显示明确的手工操作说明并退出。
- 停止器不删除数据库、上传文件或模型缓存，不停止其他 Compose 项目，也不强制关闭 Docker Desktop。

## 3. 非目标

- 不管理 Tailscale Funnel。
- 不自动安装 Docker Desktop、Git、Python、Node.js 或项目依赖。
- 不修改 `deploy/.env`，不读取或输出其中的密钥。
- 不自动创建、重置或输出管理员密码。
- 不执行 `docker compose down -v`、`docker volume rm`、系统级清理或其他不可恢复操作。
- 不把完整测试、生产构建或数据库迁移验收塞进每次启动流程；Compose 自身的 migrate 服务继续负责正常迁移。

## 4. 方案与文件职责

### 4.1 CMD 薄入口

`启动项目.cmd` 和 `停止项目.cmd` 只负责：

1. 将工作目录切换到 CMD 文件所在的项目根目录。
2. 使用 `powershell.exe -NoProfile -ExecutionPolicy Bypass` 调用对应 PowerShell 脚本。
3. 原样返回 PowerShell 退出码。
4. 失败时保留窗口并显示提示，便于双击用户查看原因。

CMD 不复制 PowerShell 中的检查逻辑，避免两套实现发生漂移。

### 4.2 启动核心脚本

`scripts/start-project.ps1` 提供默认启动和可选 `-Build` 参数：

```powershell
.\scripts\start-project.ps1
.\scripts\start-project.ps1 -Build
```

默认路径按以下顺序执行：

1. 从 `$PSScriptRoot` 解析项目根目录和 `deploy/docker-compose.yml`，不依赖调用者当前目录。
2. 读取系统可用物理内存；低于 2 GiB 时拒绝 Docker 重任务。
3. 检查 `deploy/.env` 是否存在；缺失时输出复制 `.env.example` 和手工编辑命令，然后退出。
4. 检查 Docker CLI；引擎不可用时，在 `docker desktop` 子命令存在的前提下尝试启动 Docker Desktop。
5. 在有限时间内轮询 `docker info`，超时后以非零退出码结束。
6. 执行 `docker compose -f deploy/docker-compose.yml up -d`；传入 `-Build` 时追加 `--build`。
7. 最长等待 180 秒，轮询 `http://127.0.0.1:8080/api/ready`，只有 HTTP 200 才视为业务就绪。
8. 输出 `docker compose ps` 摘要并打开 `http://127.0.0.1:8080`。

脚本成功返回 0；任何前置检查、Compose 启动或 ready 等待失败均返回非零值。

### 4.3 停止核心脚本

`scripts/stop-project.ps1` 执行以下步骤：

1. 从脚本位置解析同一个 Compose 文件。
2. 检查 Docker CLI 和引擎是否可访问。
3. 执行 `docker compose -f deploy/docker-compose.yml down --remove-orphans`。
4. 明确提示数据库、uploads 和 Hugging Face 缓存卷已保留。

停止脚本不得使用 `-v`，不得删除卷，不得调用 `docker desktop stop`。

## 5. 错误处理与用户提示

- 所有 PowerShell 脚本设置 `$ErrorActionPreference = 'Stop'`，外部命令还要显式检查 `$LASTEXITCODE`。
- 缺少 `.env` 时提示：复制 `deploy/.env.example`、编辑必要配置、重新执行启动器。
- Docker Desktop 启动失败时提示用户手动打开 Docker Desktop，而不是持续无限等待。
- Compose 启动失败时显示服务状态，并给出以下排障命令，不输出容器环境变量：

```powershell
docker compose -f deploy/docker-compose.yml ps
docker compose -f deploy/docker-compose.yml logs --tail 100 gateway api worker postgres
```

- `/api/ready` 超时时区分“容器已启动但业务未就绪”，不能只依据 `/health` 宣称项目可用。
- 启动器不自动回滚或执行 `down`，避免启动失败时误停用户正在检查的容器；用户可显式运行停止器。

## 6. 安全与 owner 边界

- 所有 Docker 命令显式绑定 `deploy/docker-compose.yml`，不枚举或停止其他项目。
- 不读取、不回显 `deploy/.env` 内容。
- 不结束其他进程，不抢占 8080 端口；端口冲突由 Compose 返回失败并展示排障提示。
- 停止只作用于当前 Compose 项目，不删除卷。
- CMD 和 PowerShell 均不包含密码、Token、API Key 或固定生产密钥。

## 7. 测试设计

按测试先行实现，新增聚焦测试验证：

1. 两个 CMD 文件调用正确的 PowerShell 脚本，并保留失败退出码。
2. 启动脚本包含 2 GiB 内存门、`.env` 检查、Docker 引擎检查和有限等待。
3. 默认启动不强制构建，`-Build` 才追加 `--build`。
4. 成功条件使用 `/api/ready` HTTP 200，不以 `/health` 代替。
5. 停止脚本只执行本项目 Compose `down --remove-orphans`。
6. 所有启动和停止文件均不包含 `down -v`、`docker volume rm`、凭据变量或 `.env` 内容输出。
7. 四个脚本通过 PowerShell/CMD 静态契约检查；PowerShell 文件通过语法解析。

不在聚焦测试中启动完整 Docker 或执行生产构建；最终只在可用物理内存不少于 2 GiB 且无其他 owner 进程冲突时，执行一次真实启动、ready 和安全停止冒烟。

## 8. 文档与使用入口

README 增加“一键启动”章节：

```text
双击：启动项目.cmd
命令：.\scripts\start-project.ps1
重建：.\scripts\start-project.ps1 -Build
停止：停止项目.cmd
```

README 同时保留原始 Docker Compose 命令，便于排障和面试解释，不用启动器隐藏底层运行方式。

## 9. 验收标准

- 双击 `启动项目.cmd` 能调用统一启动逻辑，成功后打开项目首页。
- 缺少 `.env`、内存不足、Docker 不可用、Compose 失败和 ready 超时均给出中文可操作提示并返回非零退出码。
- PowerShell 一条命令与双击入口行为一致。
- `-Build` 可以显式重建镜像，默认启动避免每次重复构建。
- 双击 `停止项目.cmd` 后本项目容器停止，持久卷仍然存在。
- 聚焦测试、脚本语法解析和一次真实启动/停止冒烟均通过后才提交实现。
