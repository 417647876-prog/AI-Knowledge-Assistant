# 阶段 3C.1：Reranker 接受门与安全拒答设计

## 背景

阶段 3C 已完成混合候选的 BGE 重排序，但当前行为只负责“重新排名”，不会删除所有低相关候选。
当检索只返回一个错误片段时，该片段即使 Reranker 分数很低，仍会作为第一名进入回答与引用。

真实 30 条验收中，`refusal-03` 正是这种情况；`multi-turn-06` 则是零候选，属于后续 Query
Rewrite 能力。阶段 3C.1 的目标是提高生产拒答安全性，不提前实现 3D，也不把单独修复一条案例
包装成当前 3C 的质量门通过。

## 目标

- Reranker 成功返回有效分数后，删除低于独立接受阈值的候选。
- 所有候选均不合格时，复用现有无依据回答，普通和流式接口均不调用 Chat Provider。
- 阈值默认关闭，升级后保持现有行为。
- 使用独立校准数据选择阈值，不使用固定 `stage3.jsonl` 验收集反向调参。
- 在评估报告中记录阈值、接受数量和最终重排结果，使结果可审计。

## 非目标

- 不修改 3C 的 MRR@5 相对提升至少 5% 的质量门。
- 不修改、删除或重标注 `stage3.jsonl` 中的案例。
- 不实现 Query Rewrite，不修复零候选的 `multi-turn-06`。
- 不引入 LLM 相关性裁判、远程 Reranker 或模型微调。
- 不把 BGE 原始分数解释成概率。

## 方案比较

### 方案 A：绝对分数接受门（采用）

对最终重排候选应用 `score >= min_score`。它能处理单候选误召回，逻辑简单，容易在普通、流式和
评估链路保持一致。缺点是阈值依赖模型和数据分布，必须独立校准。

### 方案 B：第一名与第二名分差

只有第一名明显高于第二名时才接受。它不适用于 `refusal-03` 这种只有一个候选的情况，因此不能
解决当前生产风险。

### 方案 C：LLM 二次相关性判定

让 Chat 模型判断候选是否可用于回答。语义能力更强，但会增加成本、延迟和新的失败路径，也会让
拒答结果受生成模型影响，超出本阶段范围。

## 配置契约

新增配置：

```python
rag_reranker_min_score: float | None = None
```

- `None`：关闭接受门，保持当前行为。
- 有限浮点数：只保留 Reranker 分数大于等于该值的候选。
- `NaN`、正负无穷必须在配置加载时拒绝。
- 该配置只在 Reranker 启用且成功返回分数时生效。

不得复用 `rag_score_threshold`。前者是 BGE CrossEncoder 分数，后者是向量 Retriever 的余弦检索
阈值，两者不在同一数值空间。

## 运行时数据流

```text
Hybrid candidate_k 候选
    -> BGE Reranker 批量评分
    -> 稳定降序并截取最终 top_k
    -> 接受门过滤 score < min_score
    -> 有候选：生成回答与引用
    -> 无候选：返回 NO_EVIDENCE_ANSWER、空引用、数量 0
```

接受门放在 Reranker 成功之后、回答生成之前。由于重排结果已经按分数降序，先截取 Top K 再过滤
不会遗漏更低位置但分数更高的候选。

建议增加纯函数：

```python
def accept_reranked_chunks(
    chunks: list[RetrievedChunk],
    *,
    min_score: float | None,
) -> list[RetrievedChunk]: ...
```

该函数只负责接受判断；`rerank_chunks` 继续负责 Provider 调用、分数校验和稳定排序，RagService
继续负责回退与回答编排。

## 回退语义

必须区分两类情况：

1. Provider 执行失败：仍按 `rag_reranker_allow_fallback` 决定使用原融合顺序或抛错。
2. Provider 成功但全部低分：这是有效业务判断，必须安全拒答，不能回退到原候选。

否则会出现“Reranker 已判定全部不合格，系统又用原候选回答”的安全绕过。

全候选被过滤时可写一条结构化日志，只包含 Provider 名、阈值、接受/拒绝数量和 request ID；不得
记录问题全文、片段内容或原始分数列表。

## 独立阈值校准

新增独立校准集，保存已标注的“问题—文档片段”正负样本，不复用 `stage3.jsonl` 的原问题文本。
样本来自现有测试文档，至少覆盖制度编号、语义改写、明显无关和容易误召回四类。

校准工具使用真实 `BAAI/bge-reranker-base` 批量评分，并在所有有限分数的中点中搜索阈值：

- 首要约束：负样本错误接受率为 0。
- 次要约束：正样本接受率至少 80%。
- 在满足约束的阈值中选择最低值，尽量保留更多正确候选。
- 如果不存在同时满足两项约束的阈值，校准失败，不推荐启用接受门。

校准报告记录模型名、设备、数据哈希、正负样本数量、分数范围、推荐阈值、错误接受率和正样本
接受率；不得记录密钥、连接串或用户数据。

推荐阈值只作为显式环境配置输入，不在应用启动时动态重新计算。

## 评估与质量边界

`--mode rerank` 报告新增 `rag_reranker_min_score` 和最终接受数量。正式 3C.1 验证按以下顺序：

1. 使用独立校准集生成阈值，不读取 `stage3.jsonl` 的结果来选值。
2. 显式设置阈值，重新运行固定 30 条 3C 报告。
3. 检查拒答准确率、引用命中率和 Recall/MRR 是否回归。
4. 如 MRR 相对提升仍低于 5%，保持 Task 5、3C 未完成和 3D 阻塞。

按当前离散结果推算，只修复 `refusal-03` 会让 MRR 从 28/30 提升到 29/30，相对提升约 3.57%，
仍不足 5%。因此 3C.1 的成功标准是安全拒答能力正确、无指标回归和校准可复现，不是伪造原质量
门通过。

## 测试策略

- 纯函数：阈值关闭、边界等于阈值、低分删除、顺序保持、全删除。
- 配置：默认 `None`、有限浮点、拒绝 NaN/Infinity。
- RagService：部分接受、全部拒绝、低分不触发 fallback、Provider 失败仍遵循原回退配置。
- 普通问答：全拒绝时不调用 Chat Provider，返回固定无依据回答和空引用。
- 流式问答：发出检索事件后直接输出无依据回答，生成耗时为 0、引用为空。
- 评估：最终 chunks、引用、接受数量和阈值元数据来自同一条最终链路。
- 校准：固定 Fake 分数验证阈值选择、无可行阈值失败、数据哈希和安全报告字段。
- 真实验收：只在显式命令中加载本地 BGE，普通测试不得下载模型。

## 验收条件

- 默认关闭时，现有普通、流式和评估行为不变。
- 显式阈值能删除低分片段；全部删除时不调用 Chat Provider并安全拒答。
- 有效低分结果不走 fallback；Provider 错误仍保持现有严格/允许回退语义。
- 普通、流式和评估链路使用相同最终候选。
- 校准不读取或修改固定 30 条验收数据，报告可复现且不泄露敏感信息。
- 完整测试、临时空库数据库集成、Ruff 和格式检查通过。
- 原 3C MRR 门未达到前，Task 5、3C 和 3D 状态保持不变。
