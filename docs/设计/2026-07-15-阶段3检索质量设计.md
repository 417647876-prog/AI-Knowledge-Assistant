# 阶段 3：RAG 检索质量提升总体设计

## 1. 目标

阶段 3 在现有 FastAPI、PostgreSQL、pgvector 和 Vue 知识库闭环上，建立“可测量、可对比、可回退”的 RAG 质量提升链路。

本阶段不凭少量手工提问判断效果。所有检索改造都必须先建立评估基线，再通过指标证明收益。

完整处理链路为：

```text
用户问题
  -> 必要时进行问题改写
  -> 向量检索 + 关键词检索
  -> RRF 排名融合
  -> Rerank 重排序
  -> 选出最终 Top K
  -> 大模型回答并映射真实引用
```

## 2. 当前基础与前置条件

当前仓库已经具备：

- `VectorRetriever`：按知识库隔离的 pgvector 余弦检索。
- `RagService`：问题向量化、检索、Prompt 构建、回答和引用映射。
- `EmbeddingProvider` 与 `ChatProvider` 抽象及 Fake 实现。
- 单元测试、PostgreSQL 集成测试和认证权限测试。
- 512 维本地 BGE Embedding 与 DeepSeek 兼容问答接口。

阶段 3D 复用阶段 2D 已实现的流式多轮问答边界：前端按用户与知识库隔离浏览器会话历史，调用流式问答接口时只发送受限的历史消息；后端使用 `ConversationMessage`、`QuestionRewriter` 和 `RagService.stream_answer`。阶段 3 不新增服务端会话表或消息表。开始 3D 前必须确认阶段 2D 分支已合入当前阶段分支；若未同步，只暂停 3D，不阻塞 3A、3B 和 3C。

## 4. 阶段拆分

### 4.1 阶段 3A：评估数据集与质量基线

目标是在修改检索算法前，先获得可重复的质量基线。

评估数据至少包含 30 条中文样本，覆盖：

- 需要精确关键词或编号命中的问题。
- 适合语义检索的同义表达问题。
- 文档中不存在答案、应该拒答的问题。
- 依赖上文指代的多轮问题。
- 容易被多个相似文档干扰的问题。

每条评估数据保存：

- 问题文本。
- 期望命中的知识库测试数据。
- 期望文档或片段标识。
- 是否应该拒答。
- 所属评估类别。

核心指标：

- `Recall@K`：相关片段是否进入前 K 个结果。
- `MRR@K`：第一个相关结果是否足够靠前。
- 引用命中率：回答引用是否来自标注的相关来源。
- 拒答准确率：无答案问题是否正确拒答。
- 检索耗时：记录 P50 和 P95。

普通自动化测试默认使用 Fake Provider，保证免费、稳定和可重复。真实 Embedding、Rerank 或 Chat 模型的评估必须使用显式命令运行，不能混入普通测试套件。

### 4.2 阶段 3B：PostgreSQL 混合检索

保留现有向量检索，同时增加适合中文内容的关键词检索。中文文本在写入检索字段前统一分词，PostgreSQL 负责索引、过滤和排名。

检索分为两路：

1. 向量检索返回语义候选。
2. 关键词检索返回精确词、编号、专有名词候选。

两路结果使用 Reciprocal Rank Fusion（RRF）合并。RRF 只依赖排名，不直接比较向量相似度和全文检索分数，避免不同分数体系失真。

混合检索必须满足：

- 所有查询都以 `knowledge_base_id` 作为强制过滤条件。
- 候选结果去重后再截取。
- 相同输入和相同数据库状态产生稳定排序。
- 通过配置保留纯向量模式，出现质量或兼容性问题时可以回退。
- 不在本阶段引入新的向量数据库或独立搜索服务。

### 4.3 阶段 3C：Rerank 重排序

新增独立 `RerankerProvider` 边界，避免把具体模型调用写入 `RagService`。

至少提供：

- `FakeRerankerProvider`：用于稳定单元测试。
- 本地 BGE Reranker：用于真实中文相关性排序。

混合检索先返回较多候选片段，Rerank 对“问题 + 候选片段”重新评分，最后只向回答模型传递 Top K。候选数量和最终数量必须由配置控制，并满足“候选数量不小于最终数量”。

Rerank 模型不可用时，按配置执行以下一种明确策略：

- 严格模式：返回可诊断错误。
- 回退模式：使用 RRF 融合结果继续回答并记录回退。

普通请求不得静默改变策略。

### 4.4 阶段 3D：Query Rewrite 与多轮问题改写

问题改写只解决“原问题不适合直接检索”的情况，不改变用户原始问题，也不替代最终回答模型。阶段 2D 已具备基础改写能力，阶段 3D 只补充选择性触发、失败回退和评估，不重复建立历史传输或存储机制。

触发原则：

- 首轮且语义完整的问题不额外调用改写模型。
- 短问题、包含指代词或明显依赖上文的问题才触发改写。
- 只向改写模型提供受限数量和受限长度的历史消息。
- 改写后的问题只用于检索，回答阶段仍保留原始问题语义。
- 改写失败、超时或返回空文本时，自动回退到原始问题。

问题改写需要独立边界，不能把判断、Prompt 和模型调用散落在 API 路由中。

### 4.5 阶段 3E：综合评估与验收

使用同一数据集比较：

1. 纯向量检索。
2. 向量检索 + 关键词检索 + RRF。
3. 混合检索 + Rerank。
4. 混合检索 + Rerank + Query Rewrite。

验收报告记录：

- Recall@K、MRR@K、引用命中率和拒答准确率。
- P50、P95 检索耗时。
- 每种方案相对基线的提升或退化。
- 最终推荐配置与回退配置。
- 可重复执行的命令和五分钟演示步骤。

## 5. 质量门槛

| 阶段 | 最低完成标准 |
|---|---|
| 3A | 至少 30 条中文评估数据，覆盖关键词、语义、拒答、多轮和相似文档干扰 |
| 3B | 总体 Recall@5 不低于基线；关键词类问题 Recall@5 提升至少 10 个百分点 |
| 3C | MRR@5 相对 3B 提升至少 5%；引用命中率不得下降 |
| 3D | 多轮问题 Recall@5 相对未改写结果提升至少 15 个百分点；失败回退测试全部通过 |
| 3E | Recall@5 不低于 85%，引用命中率不低于 90%，拒答准确率不低于 90% |

如果真实基线证明某个门槛不合理，只能在保留原始结果、差异分析和调整理由后修改。该调整必须使用 Sol 完成分析，并记录在执行看板中。

## 6. 测试与验证策略

每个实施任务遵守小步 TDD：

1. 先编写一个能说明行为的失败测试。
2. 运行测试并确认失败原因正确。
3. 实现让当前测试通过的最小改动。
4. 运行当前测试和直接相关测试。
5. 阶段结束时运行完整单元测试、Ruff 和数据库集成测试。
6. 把验证命令和结果写入执行看板。

必须覆盖的横向风险：

- 普通用户不能检索其他用户的知识库。
- 已删除或处理失败的文档不能进入检索结果。
- Fake Provider 测试不访问外部网络。
- 模型不可用、超时、空结果和回退行为可预测。
- 纯向量回退配置始终有自动化测试保护。

任务只有在计划规定的验证全部通过后，才能标记为已完成。

## 7. Sol 与 Terra 模型分配

当前模型含义：

- `GPT-5.6-Sol`：用于复杂、跨模块和高风险编码任务。
- `GPT-5.6-Terra`：用于边界明确的日常编码任务。

每个实施 Task 只标注一个推荐模型：`Sol` 或 `Terra`。

| 任务类型 | 推荐模型 |
|---|---|
| 代码定位、评估数据、普通测试、文档同步 | Terra |
| 小范围 Provider、配置项、独立纯函数 | Terra |
| 数据库迁移、检索接口、RRF 排名融合 | Sol |
| `RagService` 主链路、Rerank、Query Rewrite、多轮上下文 | Sol |
| 边界清晰的普通测试失败修复 | Terra |
| 同一问题连续两次验证失败、跨模块疑难问题 | Sol |
| 阶段最终代码审查和指标异常分析 | Sol |

Terra 执行同一个任务连续两次未通过相同验证时，停止继续试错并切换到 Sol。切换原因必须记录在执行看板。

## 8. 低 Token 执行方式

每次开始任务只加载：

1. `docs/实施计划/阶段3执行进度.md`。
2. 当前阶段的一份实施计划。
3. 当前 Task 文件清单中的代码。
4. 当前 Task 明确列出的直接依赖接口。

默认不加载：

- 已完成阶段的完整实施计划。
- 全仓库源代码。
- 大段历史聊天记录。
- 与当前任务无关的测试文件。

每个阶段建议使用独立 Codex 任务。新任务可使用以下最小提示：

```text
按照 docs/实施计划/阶段3执行进度.md 和当前阶段计划继续执行。
只完成当前未完成 Task。
使用计划标注的 Sol 或 Terra。
通过验证后更新执行进度，不提前执行下一阶段。
```

Sol 最终审查优先读取当前阶段 Git diff、接口契约和验证结果，不默认重新读取全仓库。

## 9. 执行看板

`docs/实施计划/阶段3执行进度.md` 是第三阶段唯一进度入口，必须包含：

- 当前阶段和当前 Task。
- 当前任务推荐模型与实际使用模型。
- 总任务数、已完成数和阶段状态。
- 最近一次验证命令与结果。
- 当前阻塞项和下一步操作。
- 各阶段任务勾选清单。
- 验证证据记录。
- Sol/Terra 切换记录。

状态只允许：

- `未开始`
- `进行中`
- `阻塞`
- `已完成`

更新规则：

- 开始 Task：标记为 `进行中`，填写实际模型。
- 验证失败：保持 `进行中`；无法继续时标记为 `阻塞` 并记录原因。
- 验证全部通过：勾选 Task，记录命令、结果和提交标识。
- 阶段验收通过：当前阶段标记为 `已完成`，下一阶段才能进入 `进行中`。

## 10. 范围外能力

阶段 3 不包含：

- Celery 或其他持久化任务队列。
- 独立 Worker 和对象存储。
- 多租户、审计日志和完整监控平台。
- 更换 PostgreSQL 或 pgvector。
- Agent 或多 Agent 问答链路。
- 模型训练或微调。
- 没有评估证据支持的大规模重构。

这些能力继续保留在阶段 4。

## 11. 完成定义

阶段 3 完成需要同时满足：

- 3A 至 3E 的实施计划全部完成。
- 执行看板中的任务全部有验证证据。
- 质量门槛全部达到，或存在经 Sol 分析并记录的合理调整。
- 纯向量回退路径仍然可用。
- 用户权限隔离和引用真实性没有退化。
- README、学习笔记和阶段 3 验收文档已经同步。
