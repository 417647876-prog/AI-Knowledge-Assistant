# 阶段 4：工程化与可部署化实施计划

> **执行要求：** 使用 `executing-plans` 逐 Task 执行；除非用户明确要求，不派发子代理。所有步骤使用本文复选框跟踪，完成一个 Task 后立即运行该 Task 的测试、检查 diff 并提交。

**目标：** 把当前只能由 FastAPI 进程内处理任务的单机 RAG 项目，升级为具备 PostgreSQL 持久化任务、严格个人数据隔离、服务端会话与用量质量基础、可观测性、安全额度和完整 Docker 编排的可部署系统。

**架构：** 保持模块化单体和单 PostgreSQL 数据源。API 只负责认证、业务接口与流式问答；独立 Worker 通过 `FOR UPDATE SKIP LOCKED` 领取数据库任务，gateway 统一提供 Vue 静态页面与 `/api`、SSE 反向代理。阶段 4 只提供阶段 5 所需的后端接口与部署底座，不开发手机 H5、Tailscale Funnel 和备份脚本。

**技术栈：** Python 3.12、FastAPI、Pydantic 2、SQLAlchemy 2 asyncio、Alembic、PostgreSQL 16、pgvector、DeepSeek OpenAI 兼容接口、Vue 3、TypeScript 5.9、Nginx、Docker Compose、pytest、Vitest、Ruff。

## 全局约束

- 设计依据：`docs/设计/2026-07-15-阶段4至5工程化与免费远程访问设计.md`。
- 阶段 4 完成前不开始阶段 5；本计划不实现响应式手机 H5、Tailscale Funnel、备份恢复脚本和真实移动网络验收。
- 保持模块化单体、单 API 进程和单 Worker 进程；不引入 Redis、Celery、RabbitMQ、Kafka、Kubernetes 或微服务拆分。
- 业务数据只属于创建它的普通用户。管理员调用普通知识库、文档、任务、会话、引用、反馈和用量接口时，也必须按管理员自己的 `user_id` 过滤，不能保留角色绕过。
- 跨用户访问统一返回安全 404；只有仍有效的单知识库只读临时授权可以进入独立支持接口。
- 任务、会话、用量、观测和审计只在 PostgreSQL 中保存；任务载荷、日志、指标、审计和管理员运营接口不得复制文档正文、问题、回答、提示词、文件名或知识库名称。
- 文档和知识库软删除后默认保留 7 天；已删除内容必须立即从普通列表、检索、问答和下载中消失。
- API 或 Worker 重启后任务不能丢失；过期租约可重新领取；旧租约晚到提交不能覆盖新租约结果。
- DeepSeek 普通与流式调用都记录真实 usage；流式请求必须发送 `stream_options.include_usage=true`。连接在 usage 返回前中断时标记 `usage_unknown`，不得根据字符数伪造 Token。
- 费用使用可配置单价快照和 `Decimal` 计算；每次模型调用前预留最大费用，真实 usage 返回后结算差额，usage 缺失时保留预留费用。
- 默认额度：每用户每天 50 次问答、每天 20 个上传文件、500 MiB 有效文件；全局月度预估费用上限 20 元；管理员可按用户覆盖个人额度。
- 登录限速默认同账号 10 分钟 5 次失败，问答限速默认每用户每分钟 10 次；阶段 4 接受单进程内存限速在 API 重启后重置的边界。
- 演示 Compose 只向 `127.0.0.1` 映射 gateway；API、Worker、PostgreSQL 和内部指标端点不得映射宿主机业务端口。
- 开发环境仍能单独启动前后端，并保留可暴露 PostgreSQL 的开发 Compose；不得为了容器演示破坏现有命令。
- 普通 `pytest` 不下载模型、不访问网络、不要求 PostgreSQL；数据库用例继续使用 `integration` marker 和 `RUN_DATABASE_TESTS=1`。
- 每个 Task 严格执行：先写失败测试、确认预期失败、写最小实现、运行局部测试、运行 Ruff、检查 diff、提交。
- PowerShell 中前端命令使用 `npm.cmd`；后端命令在 `backend/` 下使用 `uv run`。
- 不删除用户现有文档和未提交改动；迁移必须原地保留阶段 1～3 数据，并同时验证 upgrade 与 downgrade。

## C# 开发者对照

- `DocumentJob + Worker` 可理解为“数据库版 Hangfire 队列 + `BackgroundService`”，但任务领取和租约由本项目自己实现。
- FastAPI 的认证依赖和 `authorization/service.py` 类似 ASP.NET Core Policy 加带所有权过滤的 Repository。
- Pydantic 请求/响应模型对应 DTO；SQLAlchemy 模型和 Alembic 迁移分别对应 EF Core Entity 与 Migration。
- `request_id`、结构化日志和审计事件类似 ASP.NET Core Middleware、`ILogger` scope 与审计表，三者职责不能混在一起。

- [x] **Step 3：实现 PostgreSQL 原子状态机**

领取事务固定使用下面的候选查询，只包括到期的 `pending`、到期的 `retry_wait` 和租约已过期的 `processing`：

```sql
SELECT id
FROM document_jobs
WHERE (status = 'pending' AND run_after <= :now)
   OR (status = 'retry_wait' AND run_after <= :now)
   OR (status = 'processing' AND lease_expires_at < :now)
ORDER BY run_after, created_at, id
FOR UPDATE SKIP LOCKED
LIMIT 1;
```

领取成功后同一事务写入唯一 UUID 租约、Worker ID、尝试次数、开始时间和心跳。所有阶段更新、完成和失败 SQL 都同时约束 `id`、`status=processing` 和 `lease_token`，受影响行数不是 1 时视为租约失效。

错误信息只允许保存稳定错误码和面向用户的脱敏消息；异常类型、连接串、文件路径和第三方响应正文只能留在受清洗的结构化日志中。
