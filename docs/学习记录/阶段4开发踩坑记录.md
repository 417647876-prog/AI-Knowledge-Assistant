# 阶段 4 开发踩坑记录

## 1. 这篇记录解决什么问题

阶段 4 不只是把功能“写出来”，而是把项目从 FastAPI 单进程演示升级为可恢复、可隔离、可观测、可容器运行的工程闭环。很多问题在单元测试或静态 Compose 检查中不会出现，必须经过并发、断线、数据库重启和真实 Docker 启动才能暴露。

下面只记录本轮实际遇到的问题。每一项都按“现象 → 根因 → 解决办法 → 以后怎么防”整理，便于复盘和面试讲解。

## 2. 持久化任务与并发

### 坑 1：FastAPI `BackgroundTasks` 不是可靠任务队列

**现象：** 文档上传接口返回成功后，只要 API 进程重启，尚未完成的解析和向量入库就可能丢失。

**根因：** `BackgroundTasks` 只存在于当前 API 进程内存，没有持久化状态、租约、重试和崩溃恢复能力。

**解决：** 将任务写入 PostgreSQL，由独立 Worker 使用 `FOR UPDATE SKIP LOCKED` 原子领取；任务保存租约、心跳、尝试次数和稳定错误码。API 只入队，不再直接承担耗时入库。

**C# 对照：** 这类似把 Controller 中的 `Task.Run()` 改成数据库 Outbox/Job Table 加独立 `BackgroundService`。HTTP 请求结束不等于后台任务完成。

### 坑 2：租约过期后，旧 Worker 的“晚到结果”可能覆盖新任务

**现象：** Worker A 超时后任务被 Worker B 重新领取；A 随后恢复，如果仍能提交结果，就可能覆盖 B 的新状态或重复替换切片。

**根因：** 只按任务 ID 更新状态，没有同时校验当前租约持有者和租约版本。

**解决：** 完成、失败和切片替换都必须携带当前租约条件；租约已过期或已换持有者时，晚到提交直接忽略。切片替换与任务完成放在同一事务中。

**以后怎么防：** 并发测试必须覆盖两个 Worker 竞争、租约过期、重复执行和晚到提交，不能只测单 Worker 正常路径。

### 坑 3：时间相关测试容易偶发失败

**现象：** 租约测试单独运行通过，全量运行时偶尔在过期边界失败。

**根因：** 测试混用了数据库时间、应用当前时间和真实等待，断言正好落在时间边界。

**解决：** 固定测试时间基准，显式传入 `now`，避免用 `sleep` 猜测租约是否过期。

### 坑 4：重试错误不能直接保存异常全文

**现象：** 把原始异常写入任务错误字段很方便，但异常可能包含文件路径、数据库连接信息、文档内容或模型响应。

**根因：** 把“可诊断”误解为“保存所有异常文本”。

**解决：** 任务只保存服务端固定错误码和必要元数据，重试策略使用白名单；未知异常进入脱敏日志，不复制到业务表。

## 3. 权限、生命周期与事务

### 坑 5：管理员角色不等于可以绕过内容所属人

**现象：** 早期权限逻辑容易写成“管理员可以查询所有知识库”，这会破坏个人文档的严格隔离。

**根因：** 把“管理账号和运营元数据”与“读取用户正文”混成同一种管理员能力。

**解决：** 普通知识库、文档、任务、会话、引用和用量接口始终按当前 `user_id` 过滤，管理员也不例外。排障使用单独的支持授权：限知识库、限管理员、只读、可撤销、会过期并且留审计。

**C# 对照：** `[Authorize(Roles = "Admin")]` 只说明能进入管理员端点，不代表 EF Core 查询可以去掉 `OwnerId` 条件。

### 坑 6：支持授权的“检查后使用”存在竞态

**现象：** 请求先查询授权有效，随后授权被撤销或次数被其他请求消耗，但当前请求仍继续读取。

**根因：** 授权检查与消费分成两个事务，没有锁定同一行。

**解决：** 在一个事务内锁定授权、校验有效期/撤销状态并原子消费；失败统一返回安全结果，不泄露资源是否存在。

### 坑 7：软删除、恢复和永久清理是三套边界

**现象：** 仅增加 `deleted_at` 后，已删除文档仍可能参与检索；恢复时可能与同名资源冲突；永久清理还可能误删上传根目录外文件。

**根因：** 把“列表中看不见”误认为“整个系统已删除”。

**解决：** 所有检索入口排除已删除资源；恢复显式处理冲突；永久清理由 Worker 执行，校验保留期限和规范化路径，拒绝根目录外路径及符号链接逃逸，并记录固定审计码。

### 坑 8：流式断线时，消息、Token 和费用容易只完成一半

**现象：** SSE 客户端中途断开后，可能出现回答消息已写入但用量未结算，或预留费用未释放。

**根因：** 流式输出跨越较长时间，客户端生命周期与数据库事务生命周期不同；锁顺序不一致还可能造成死锁。

**解决：** 明确回答状态机和统一锁顺序；断线也执行收尾事务；Token 用量分为完整、未知等状态，不能用 `0` 假装已统计。

### 坑 9：额度检查必须抵抗并发，不能“先查后加”

**现象：** 两个并发请求都看到剩余额度足够，然后同时调用外部模型，最终越过每日或全局费用上限。

**根因：** 普通的 `SELECT` 后再 `UPDATE` 不是原子预留。

**解决：** 在调用模型前通过数据库锁和费用预留完成原子检查；结算时再按真实 usage 调整。测试还要固定资源创建/清理顺序，避免外键残留掩盖真正的并发问题。

### 坑 10：日志和审计的“标准字段”也可能泄密

**现象：** 即使不主动记录正文，异常消息、日志参数、审计摘要和管理员聚合字段仍可能带入问题、回答、文件名或 Token。

**根因：** 只检查日志正文，没有递归扫描结构化字段；审计为了排障接受了任意字符串。

**解决：** 对日志、指标、审计和运营响应做敏感键扫描；审计摘要只接受固定枚举和数字。最终只给永久清理放行 `PURGE_PATH_INVALID`、`PURGE_RETENTION_ACTIVE` 两个服务端错误码，没有放宽客户端文本。

## 4. Docker、网关与真实运行

### 坑 11：`/health` 成功不代表服务已经可用

**现象：** API 进程能返回 `/health`，但数据库未连接或 Alembic 迁移未到 head，真实业务请求仍失败。

**根因：** 存活检查和就绪检查混用。

**解决：** `/health` 只表示进程活着；`/ready` 必须验证 PostgreSQL 和当前迁移版本。gateway、Compose 和验收脚本都检查经过代理的 `/api/ready`。

### 坑 12：只读容器里继续使用 `uv run` 会失败

**现象：** 静态镜像构建通过，容器启动后 `uv run` 尝试写缓存或同步开发依赖，在只读根文件系统中报错。

**根因：** 把开发环境命令原样带进运行时容器。

**解决：** 构建阶段安装依赖，运行阶段直接调用镜像 PATH 中的 `alembic`、`uvicorn` 和 `python`。运行镜像只包含必要脚本，例如管理员初始化入口。

### 坑 13：Nginx 配置正确不等于静态页面和宿主端口都可用

**现象：** 覆盖默认模板后健康接口正常，但 SPA 静态文件找不到；gateway 只连接 Docker internal 网络时，Docker Desktop 又不会建立宿主端口转发。

**根因：** 自定义模板遗漏 `root/index`；同时误以为 internal 网络仍允许宿主端口发布。

**解决：** 显式配置 `/usr/share/nginx/html`、`index.html`、SPA fallback 和 SSE `proxy_buffering off`；只让 gateway 同时连接公共 bridge 与内部固定网段，其余服务保持内部网络且不发布端口。

### 坑 14：静态 Compose 测试发现不了真实运行时缺件

**现象：** YAML、Dockerfile 和 Nginx 契约测试都通过，真实启动仍暴露缺少 `uv`、初始化脚本、错误 Origin 或 gateway 信任链不完整等问题。

**根因：** 静态测试只能证明配置形状，不能证明镜像里真的能导入模块、执行迁移和完成登录/SSE。

**解决：** 保留静态契约测试，同时增加真实 `compose up`、管理员 CLI、认证业务脚本和容器重启测试。最终宿主机只允许 `127.0.0.1:8080`。

### 坑 15：PostgreSQL 重启后，旧 Worker 心跳会造成假通过

**现象：** 重启测试查询到一条“未过期”心跳就判断 Worker 已恢复，但这条记录可能是数据库重启前留下的。

**根因：** 只比较最大允许年龄，没有比较“数据库重启完成时间”。

**解决：** 在 PostgreSQL restart 返回后记录时间，要求新心跳严格晚于该时间；Worker 还要捕获数据库暂时不可用的 SQLAlchemy 异常，恢复后继续循环，而不是进程直接退出。

### 坑 16：Docker 构建失败不一定是代码问题

**现象：** BuildKit 出现 I/O 错误、VHDX 只读或构建莫名中断，反复改 Dockerfile 也无效。

**根因：** C 盘被构建缓存和 Docker VHDX 写满，同时并行构建造成内存压力。

**解决：** 先检查磁盘和物理内存，再排查代码；只清理未使用的 BuildKit 缓存和重复镜像，校验后把 Docker 数据迁移到 G 盘。重任务串行执行，开始前至少保留 2 GiB 可用物理内存。

**以后怎么防：** 每次 Docker 大构建前先记录 C/G 盘余量、VHDX 位置、`docker system df` 和物理内存；不要把 `docker compose down -v` 当普通停止命令。

## 5. 全量验收与文档

### 坑 17：失败后的测试数据库不能继续当“干净环境”

**现象：** 第一次集成测试失败后直接重跑，出现越来越多外键残留和无关失败，难以区分首因与连锁反应。

**根因：** Task 11 新增 `user_quotas` 后，旧测试清理器没有同步删除；失败运行又污染了共享测试库。

**解决：** 补齐 `UserQuota` 清理；全量集成使用不挂载开发卷的一次性 PostgreSQL 容器，失败后销毁并重新创建，绝不在持久开发库上执行降级测试。

### 坑 18：评测环境真实，不等于所有模型都是真实 Provider

**现象：** 报告最初容易把阶段 3 复验写成“真实质量回归”，但本轮 Chat Provider 实际为 Fake。

**根因：** 混淆了真实 PostgreSQL、本地 BGE 检索与真实 DeepSeek 回答三个不同层次。

**解决：** 文档明确写成“真实 PostgreSQL + 本地 BGE Embedding/Reranker + Fake Chat”，指标证明固定检索、引用和门禁编排没有退化，但不代表真实 DeepSeek 回答质量。

### 坑 19：测试数量多不等于证据归属准确

**现象：** 阶段 4 报告曾把 `test_structured_logging.py` 归入 204 项 PostgreSQL 集成测试，但它实际属于普通单元回归。

**根因：** 收口时只记住总数，没有核对测试目录和门禁来源。

**解决：** 最终报告分别记录：625 项普通回归、204 项 PostgreSQL 集成、170 项前端测试、20 步 Compose 业务脚本和 1 项重启恢复测试。

## 6. 最值得保留的工程经验

1. **先区分存活、就绪和业务可用。** 一个 200 响应不能证明整个系统正常。
2. **并发正确性依赖数据库原子条件。** Python `if`、C# `lock` 或单进程内存状态都挡不住多进程竞争。
3. **管理员权限也要最小化。** 运营权限与用户正文访问权限必须分开建模。
4. **静态测试、集成测试和真实运行各自证明不同事情。** 三者不能互相替代。
5. **先检查资源，再怀疑代码。** 磁盘写满、VHDX、内存和网络拓扑都可能表现成“代码构建失败”。
6. **验收报告也需要审查。** Provider、测试数量、证据路径和已接受边界写错，会让技术上通过的项目在面试中失去可信度。
