# 阶段 3 验证与演示

## 1. 前置条件

- Docker PostgreSQL 已启动，目标知识库已导入固定的 5 份评估文档。
- 后端环境能够使用本地 `BAAI/bge-small-zh-v1.5`、本地 `BAAI/bge-reranker-base` 和真实聊天模型。
- 固定数据集为 `backend/tests/fixtures/evaluation/stage3.jsonl`，包含 30 条关键词、语义、拒答、多轮和干扰案例。
- `backend/reports/*.json` 是被 Git 忽略的本地产物；正式 Markdown 报告可以提交。
- 密钥和数据库连接只从本地环境加载，不复制到命令、报告或演示材料。

## 2. 一键验收命令

在本地环境已加载的 PowerShell 中执行：

```powershell
Set-Location (git rev-parse --show-toplevel)
Set-Location backend
$env:STAGE3_KNOWLEDGE_BASE_ID = "准备好的评估知识库 UUID"
uv run python -m scripts.accept_stage3 `
  --dataset tests/fixtures/evaluation/stage3.jsonl `
  --knowledge-base-id $env:STAGE3_KNOWLEDGE_BASE_ID `
  --policy config/evaluation/stage3-quality-policy.json `
  --reports-dir reports `
  --markdown-output ..\docs\验收与演示\阶段3质量验收报告.md
$acceptanceExitCode = $LASTEXITCODE
Remove-Item Env:STAGE3_KNOWLEDGE_BASE_ID
Write-Output "accept_stage3 exit code: $acceptanceExitCode"
```

脚本按 `vector → hybrid → rerank → rewrite` 固定顺序运行，四种模式共用同一 run ID 和知识库快照。不要通过命令行临时覆盖质量阈值。

## 3. 验收产物

- `backend/reports/stage3e-vector.json`：纯向量基线。
- `backend/reports/stage3e-hybrid.json`：向量与关键词混合检索。
- `backend/reports/stage3e-rerank.json`：混合候选经过本地 BGE 重排。
- `backend/reports/stage3e-rewrite.json`：选择性问题改写后的最终推荐模式。
- `backend/reports/stage3e-manifest.json`：五个公开产物的 SHA-256、12 个质量门状态、run ID 和快照摘要。
- `docs/验收与演示/阶段3质量验收报告.md`：固定 12 节的脱敏中文报告。

manifest 最后写入。演示时先核对它，再打开正式报告；若中途写入失败，旧 manifest 不会被半套新文件覆盖。

## 4. 五分钟演示时间线

### 0:00～0:40：问题与评估集

说明企业知识库检索不能只凭“回答看起来不错”验收。展示固定 30 条数据的五类分布，以及 Recall@5、MRR@5、引用命中率、拒答准确率和延迟指标。

### 0:40～1:30：vector 到 hybrid

展示关键词编号与中文精确词容易被纯语义向量漏掉。说明 PostgreSQL 关键词检索与向量检索通过 RRF 融合，真实综合 Recall@5 从 83.33% 提升到 93.33%。

### 1:30～2:15：rerank 与 3C 风险豁免

说明本地 BGE 对混合候选重新排序，但固定数据中 hybrid 已有大量第一名结果，rerank 的 MRR 相对提升为 0.00%，没有达到至少 5% 的原质量门。本阶段保留“质量门未通过、已获风险豁免”的事实；只有 MRR 不负增长且引用不下降时豁免才适用。

### 2:15～3:10：rewrite 与安全回退

展示只有需要上下文的多轮追问才改写。改写后的独立问题用于检索，回答 Prompt 仍保留用户原问题；精确的 `QUESTION_REWRITE_ERROR` 会回退到原问题，其他错误不会被吞掉。

### 3:10～4:10：同快照四模式与最终质量门

打开正式报告的四模式表和 gate 表。强调四份报告使用同一数据集、Top K、案例顺序、run ID 和知识库快照。最终只用 rewrite 的三项绝对门收口：Recall@5 96.67%、引用命中率 100.00%、拒答准确率 96.67%，分别达到 85%、90%、90% 的目标。

### 4:10～5:00：manifest、防泄漏与回退

展示 manifest 中的五个哈希和 12 个 gate 状态，再说明 Markdown 只展示 run ID、12 位快照摘要和失败案例 ID，不包含完整知识库 UUID、问题全文、片段全文、连接串或密钥。最终推荐 rewrite；故障隔离时回退 vector。

## 5. 现场问题

现场使用已准备的公开测试问题，不复制知识库私密片段：

1. 编号关键词：询问制度中的明确编号或缩写，观察 hybrid 对精确词的补充。
2. 语义问题：换一种说法询问同一制度，观察本地 Embedding 的语义召回。
3. 多轮指代：先询问一个制度，再追问“那它能顺延吗”，观察选择性改写。
4. 无答案拒答：询问固定文档未覆盖的内容，确认系统安全拒答且不伪造引用。

演示只展示问题、指标和结构化引用文件名；不展示检索片段全文或完整 Prompt。

## 6. 失败演示

- 退出码 `0`：所有非豁免质量门通过，产物完整。
- 退出码 `1`：输入、环境、schema、快照一致性或写盘错误。修复原因后整套重跑，不直接编辑报告。
- 退出码 `2`：存在未豁免的质量门失败。保留本轮产物用于诊断，不修改固定数据集或策略阈值制造通过。

失败演示可以使用无效的临时输出路径或测试替身说明退出码，不删除开发数据库、不修改正式策略，也不破坏已有 manifest。

## 7. 回退配置

生产故障隔离时使用既有配置切回纯向量：

```powershell
$env:RAG_RETRIEVAL_MODE = "vector"
$env:RAG_RERANKER_PROVIDER = "disabled"
```

该回退减少本地 Reranker 和问题改写依赖，不需要改变数据库 schema。问题排除后，再按完整质量验收流程恢复 rewrite 推荐链路。
