# 阶段 2A：Vue 前端知识库工作台设计

## 1. 目标与边界

阶段 2A 为既有 FastAPI RAG 后端提供一个可直接操作的单页管理台。用户可以创建和选择知识库、上传文档、观察文档处理状态、发起提问并查看回答与引用。

本阶段不实现注册登录、JWT、权限隔离、流式输出、聊天历史、文档删除或修改后端 API。这些能力属于后续 2B 至 2D。

## 2. 技术选择

- Vue 3：组合式 API 组织页面与组件。
- Vite：本地开发服务器和生产构建。
- TypeScript：为后端请求和响应定义静态类型。
- Element Plus：提供表单、表格、上传、提示、空状态与加载组件。
- Pinia：保存当前知识库、文档列表和加载状态。

前端使用浏览器的 `fetch` 调用当前 FastAPI，不引入额外 BFF 或 Node 服务。开发环境通过 Vite 代理将 `/api`、`/health` 和 `/ready` 转发到 `http://127.0.0.1:8000`，避免浏览器跨域问题。

## 3. 页面与组件

首页为单页工作台，固定为两栏布局：

```text
┌────────────────┬─────────────────────────────────────┐
│ 知识库侧边栏   │ 工作区                                │
│ - 知识库列表   │ - 当前知识库名称与说明               │
│ - 新建知识库   │ - 文档上传                           │
│                │ - 文档处理状态表                     │
│                │ - 问答输入、回答和引用               │
└────────────────┴─────────────────────────────────────┘
```

组件职责：

- `KnowledgeBaseSidebar`：读取知识库列表、选择当前知识库、弹窗创建知识库。
- `DocumentUpload`：校验文件扩展名，调用上传接口，并将新文档加入状态表。
- `DocumentTable`：展示文件名、状态、错误信息和处理中的加载状态。
- `QuestionPanel`：输入问题、提交问答请求、展示回答和结构化引用。
- `App`：仅承担布局与初始加载，不处理具体接口细节。

## 4. API 与状态设计

`src/api/` 按后端资源拆分：

- `knowledgeBases.ts`：`GET/POST /api/v1/knowledge-bases`。
- `documents.ts`：上传文档和查询 `GET /api/v1/documents/{document_id}`。
- `questions.ts`：调用 `POST /api/v1/knowledge-bases/{knowledge_base_id}/questions`。

`src/types/api.ts` 定义知识库、文档任务、问答响应和引用类型。所有 API 模块统一把 FastAPI 的错误信封转换为包含状态码、错误码、中文提示和 `request_id` 的前端错误对象。

Pinia store 保存 `knowledgeBases`、`activeKnowledgeBaseId`、`documents` 与页面加载状态。选择知识库后只刷新该知识库在当前页面会话上传或查询过的文档；后端目前没有文档列表接口，因此不能伪造“完整历史文档列表”。

## 5. 关键交互

1. 页面加载时获取知识库列表；没有知识库时引导用户创建。
2. 选择知识库后显示上传与问答区域；未选择时禁用二者。
3. 上传成功后立即显示 `pending`，每 2 秒轮询文档状态；变为 `ready`、`failed` 后停止轮询。
4. 文档未处理完成时仍允许提问，但页面提示结果可能不包含新文档内容。
5. 问答成功时显示答案、检索片段数量、文件名、页码/工作表和相关片段。
6. 系统拒答时显示后端原始拒答文案，不额外编造解释。

## 6. 错误处理与可用性

- 请求期间禁用重复提交按钮并显示加载状态。
- `409`、`413`、`415` 和解析失败均使用 Element Plus 中文消息提示。
- 网络不可用或 `5xx` 时提示“服务暂不可用”，并展示可复制的 `request_id`（若响应包含）。
- 表单中限制知识库名称和问题不能为空；文件格式限制与后端保持一致，但以后端校验结果为准。

## 7. 验证策略

- 单元测试：API 错误映射、文档状态轮询的停止条件和超时条件。
- 组件测试：创建成功后的列表刷新、上传后状态展示、失败状态提示、问答引用渲染。
- 手工验收：在 Docker + FastAPI 环境中，使用第一阶段测试资料完成创建、上传、等待、提问和引用检查。
- 构建检查：TypeScript 类型检查、生产构建和前端测试均必须通过。

## 8. 完成标准

- 浏览器能在本地通过 Vite 代理调用当前 FastAPI。
- 用户无需 Swagger 即可完成创建知识库、上传支持文件、查看状态和提问。
- 已处理文档的回答能展示后端返回的真实引用。
- 后端返回的常见错误能以中文提示展示，且不丢失错误码和请求标识。
- 前端测试、类型检查和生产构建通过。
